## The 4 items

1. **`type` should describe CONTENT, not provenance.** AGREE (strongly).
   - Brent's reasoning: OKF's point is portability; dropped into another KB, `type: daydream`
     is nonsensical. Provenance ‚âÝ content type.
   - Support: provenance is ALREADY carried by `x_source: daydream`, so `type` is free to be
     content-descriptive with zero info loss. Per OKF SPEC, `type` classifies the knowledge.
   - Open design Q (the only nuance): what content taxonomy? Freeform (e.g. "issue",
     "bug-fix", "procedure", "fact", "reference") vs a small controlled vocab. Decide before
     implementing; a controlled vocab helps routing/retrieval but needs the extractor to map.

2. **`title` should be human-readable.** AGREE. Currently == `item_id`. Want a real title
   (e.g. "pytest --pastebin lexer=python3 ‚Üí HTTP 400"). Generated by the daydream extractor.

3. **`description` should be a real description, not a truncated slice.** AGREE. Currently a
   mechanical prefix of the body ending in `‚Ä¶`. Want an actual summary (extractor-generated).

4. **Kill duplication.** AGREE, with one nuance.
   - `x_item_id` / `title` / `resource` all encode the id. NOTE: fixing #2 (human title)
     already DECOUPLES title from id, resolving part of this. Then prefer keeping `resource`
     (OKF-standard URI, the canonical id bearer) and drop `x_item_id` IF the id is recoverable
     from `resource` (`memeval://memory/<id>`) at load ‚Äî verify the loader can parse it.
   - `x_metadata_json: {"extracted_from": <session>}` duplicates `x_session_id` for daydream
     memories (extracted_from IS the session). Drop `x_metadata_json` when it carries nothing
     beyond session id; keep it only when it holds genuinely new metadata.
